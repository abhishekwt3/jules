01The Problem

Every accounting tool I'd used shipped a heavy backend just to add two numbers and make sure they balanced. For small apps, indie tools, and offline-first products, that's a tax — a server, a database, a migration story — before you've recorded a single transaction.

I wanted a ledger you could drop into any app: correct by construction, fast enough to feel instant, and small enough to run entirely client-side.

"It should be impossible to write books that don't balance — the type system, not the test suite, should enforce it."

02The Approach

I built the core engine in Rust, modelling accounts, entries, and postings as types where an unbalanced transaction simply doesn't compile into a valid state. The engine compiles to a ~90KB WASM module with a thin TypeScript SDK on top, so it feels like calling any other JS library.

Persistence is pluggable — IndexedDB by default, but anything that can store bytes works. The whole thing runs in a web worker so the UI never blocks, even when replaying years of transactions.

03The Outcome

Ledger now powers a handful of indie finance apps and crossed 6,000 GitHub stars in its first year. The thing I'm proudest of isn't the stars — it's the issues that aren't filed. Nobody reports unbalanced books, because they can't happen.
